  <files>src/app/api/recipes/route.ts, src/app/api/recipes/[id]/route.ts</files>
  <action>
**POST /api/recipes/route.ts:**
- Destructure `recipeIngredients` from request body (optional array)
- Filter out invalid rows: skip where `ingredientId` is falsy or `amount` is NaN/undefined
- Add `recipeIngredients: { create: validRows.map(row => ({ ingredientId: row.ingredientId, amount: row.amount, unit: row.unit })) }` to `prisma.recipe.create` data
- Add `recipeIngredients: { include: { ingredient: { select: { id: true, name: true, kcalPer100g: true, proteinPer100g: true, fatPer100g: true, carbsPer100g: true } } } }` to include clause
- If `recipeIngredients` is undefined/empty, just don't include the create — existing behavior unchanged

**GET /api/recipes/[id]/route.ts:**
- Add `recipeIngredients: { include: { ingredient: { select: { id: true, name: true, kcalPer100g: true, proteinPer100g: true, fatPer100g: true, carbsPer100g: true } } }, orderBy: { createdAt: "asc" } }` to the findUnique include clause
- This makes the modal and detail page automatically receive nutrition data

**PUT /api/recipes/[id]/route.ts:**
- Destructure `recipeIngredients` from request body (optional array)
- If `recipeIngredients` is provided (even empty array), use `prisma.$transaction()`:
  1. `prisma.recipeIngredient.deleteMany({ where: { recipeId: id } })`
  2. `prisma.recipe.update({ where: { id }, data: { ...recipeFields, recipeIngredients: { create: validRows } }, include: { creator: ..., recipeIngredients: { include: { ingredient: ... } } } })`
- If `recipeIngredients` is undefined (not provided), do existing update without touching recipeIngredients — backward compatible
- Filter invalid rows same as POST (skip empty ingredientId or NaN amount)
- Add recipeIngredients to include clause on the update response

**Important:** Do NOT populate estimatedKcal/Protein/Fat/Carbs fields — those are Phase 9 only.
